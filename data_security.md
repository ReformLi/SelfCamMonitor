# 数据安全规划

---

## 一、当前安全状态

### 1. 视频推流

- **协议**：HTTP + MJPEG，无加密。
- **认证**：可选 Basic Auth，但凭据以 Base64 明文传输。
- **风险**：同局域网内任意设备可监听视频流及密码。

### 2. 报警推送

- **方式**：HTTP POST JSON 至用户配置的 URL。
- **内容**：包含事件时间戳、消息等，但无敏感个人信息。
- **风险**：若 URL 为内网 HTTP，同样可被嗅探；若为公网 HTTPS，传输相对安全但依赖目标服务器。

### 3. 录像存储

- **位置**：应用专属目录，卸载即删，无额外权限保护。
- **访问**：仅本应用及系统可访问，其他应用无法读取。

### 4. 网络暴露

- **端口**：8080 固定监听，无 IP 白名单或访问控制（除 Basic Auth）。
- **威胁**：若手机暴露在公网（如端口转发），易被暴力破解或漏洞利用。

### 5. 应用安全

- **权限**：相机、录音、前台服务等，均为必需。
- **代码**：无混淆，可能泄露敏感逻辑或硬编码字符串。

## 二、风险矩阵

| 风险            | 严重程度 | 可能性        | 影响             |
| ------------- | ---- | ---------- | -------------- |
| 视频流/密码被局域网嗅探  | 中    | 高（同一WiFi）  | 隐私泄露，摄像头被未授权观看 |
| 报警信息被窃听       | 低    | 高          | 仅事件通知，无实时画面    |
| 摄像头端口被公网扫描    | 中    | 低（需手动端口转发） | 未授权访问或拒绝服务     |
| 认证凭据暴力破解      | 中    | 低          | 通过公网暴力破解后获取视频流 |
| APK 反编译泄露代码逻辑 | 低    | 低          | 暴露内部实现，可能被寻找漏洞 |
| 录像文件被物理窃取     | 低    | 低          | 需获取设备文件系统访问权   |

## 三、安全增强规划（按优先级）

### 🔴 高优先级（建议立即实施）

#### 1. 视频流加密（HTTPS）

- **方案**：内建自签名证书，启动时自动生成，浏览器需信任证书（或显示不安全）。
- **替代方案**：使用 **WebSocket Secure (WSS)** 代替 MJPEG，或升级为 WebRTC（自带 DTLS-SRTP 加密）。  
  - 短期简易实现：在 NanoHTTPD 前挂接一个 `SSLServerSocket`，加载自签名 keystore。
- **收益**：防止局域网嗅探，保护密码和画面。

#### 2. HTTP Basic Auth 升级为 Digest Auth 或令牌认证

- **方案**：
  - 实现 Digest Auth（NanoHTTPD 无内置，需自行实现）。
  - 更佳：使用 **Bearer Token**，用户登录后获得 token，所有请求携带 token。
- **当前缓解**：若启用 HTTPS，Basic Auth 的明文问题可缓解。

#### 3. 本地录像加密

- **方案**：在 MediaMuxer 写入文件时，使用 `CipherOutputStream` 加密，播放时解密。
- **简易版**：使用 Android `EncryptedFile`（Jetpack Security）包装文件。
- **收益**：即使手机丢失或文件被导出，录像内容不可读。

### 🟡 中优先级（建议近期实现）

#### 4. 端口访问控制

- **方案**：仅监听局域网地址（`0.0.0.0` 改为实际 WiFi IP），不监听公网。
- **增加 IP 白名单**：允许用户在设置中指定可访问的 IP 段。
- **实现**：在 `serve()` 中检查请求来源 IP，拒绝非白名单连接。

#### 5. 报警 HTTPS 强制

- **方案**：在 AlertManager 中，若用户输入的 URL 为 HTTP 且非本地地址，弹出警告或拒绝。
- **建议**：强制使用 HTTPS URL。

#### 6. 登录限速与锁定

- **方案**：连续认证失败 5 次后，锁定该 IP 5 分钟。
- **实现**：在 `StreamServer` 中维护一个失败计数器 Map。

### 🟢 低优先级（可后续迭代）

#### 7. 应用混淆与加固

- **方案**：在 `build.gradle` 中启用 `minifyEnabled true` 并使用 ProGuard 规则。
- **收益**：增加反编译难度，保护核心逻辑。

#### 8. 安全审计日志

- **方案**：记录认证失败、IP 拒绝等事件到本地文件，便于事后追溯。
- **注意**：日志本身不应包含敏感数据。

#### 9. 摄像头物理遮挡提醒

- **方案**：检测到摄像头未被遮挡时可发送提醒（需额外传感器或图像分析）。

## 四、推荐安全增强包（最小可行）

针对家用环境，建议尽快实施以下三项，能覆盖 80% 的风险：

1. **HTTPS 自签名证书**（保证通信加密）。
2. **IP 白名单或只监听局域网**（防止公网暴露）。
3. **录像文件加密**（保护隐私）。

---

## 五、实施路线图

- **v4.1**：添加 HTTPS 支持（自签名），视频流和认证全部加密。
- **v4.2**：增加登录失败锁定及 IP 白名单设置。
- **v4.3**：启用 Jetpack Security 加密录像文件。
- **v4.4**：代码混淆与安全加固。

以上规划可根据你的实际需求和用户规模逐步推进。当前项目作为内网监控，Basic Auth + 专用 WiFi 已能满足基本安全需求，但如果要在更复杂的网络环境（如合租、公网）使用，强烈建议优先升级到 HTTPS。
